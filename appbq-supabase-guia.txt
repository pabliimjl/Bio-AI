AppBQ + Supabase (guia practica para backend gratis)

Objetivo
- Usar Supabase como backend administrado sin montar servidor propio.
- Mantener frontend simple (index.html + script.js + style.css) y escalar despues.

1) Arquitectura recomendada

Frontend (tu app actual)
- index.html
- style.css
- script.js

Backend administrado (Supabase)
- PostgreSQL (tablas)
- Auth (usuarios)
- Storage (imagenes/archivos)
- API REST automatica
- Realtime opcional

2) Estructura minima de proyecto

AppBQ/
  index.html
  style.css
  script.js
  supabase-config.js        # cliente de Supabase
  README.md

3) Configuracion en Supabase (paso a paso)

1. Crear proyecto en Supabase.
2. Ir a SQL Editor y crear tablas base.
3. Activar RLS (Row Level Security).
4. Crear policies por tabla.
5. Crear bucket en Storage si subes imagenes.
6. Copiar:
   - SUPABASE_URL
   - SUPABASE_ANON_KEY
7. Configurar esas variables en tu frontend.

4) Esquema SQL inicial sugerido

Tabla de perfiles (1 a 1 con auth.users):

create table if not exists profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  nombre text,
  creado_en timestamptz default now()
);

Tabla principal de ejemplo (items/pedidos/lo que maneje AppBQ):

create table if not exists registros (
  id bigint generated by default as identity primary key,
  user_id uuid not null references auth.users(id) on delete cascade,
  titulo text not null,
  descripcion text,
  estado text default 'activo',
  creado_en timestamptz default now()
);

5) Politicas RLS recomendadas (seguridad)

alter table profiles enable row level security;
alter table registros enable row level security;

-- profiles: cada usuario solo ve/edita su perfil
create policy "profiles_select_own"
on profiles for select
using (auth.uid() = id);

create policy "profiles_update_own"
on profiles for update
using (auth.uid() = id);

create policy "profiles_insert_own"
on profiles for insert
with check (auth.uid() = id);

-- registros: cada usuario maneja solo sus datos
create policy "registros_select_own"
on registros for select
using (auth.uid() = user_id);

create policy "registros_insert_own"
on registros for insert
with check (auth.uid() = user_id);

create policy "registros_update_own"
on registros for update
using (auth.uid() = user_id);

create policy "registros_delete_own"
on registros for delete
using (auth.uid() = user_id);

6) Cliente en frontend (supabase-config.js)

import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';

const SUPABASE_URL = 'https://TU-PROYECTO.supabase.co';
const SUPABASE_ANON_KEY = 'TU_ANON_KEY';

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY);

7) Flujos base en script.js

Autenticacion:
- Registro: supabase.auth.signUp({ email, password })
- Login: supabase.auth.signInWithPassword({ email, password })
- Logout: supabase.auth.signOut()
- Sesion actual: supabase.auth.getSession()

CRUD de registros:
- Crear: insert en registros con user_id del usuario logueado
- Listar: select * desde registros (RLS ya filtra por user)
- Editar: update por id
- Eliminar: delete por id

Upload de imagenes (opcional):
- Bucket: appbq-assets
- Ruta recomendada: user_id/nombre-archivo
- Subir: supabase.storage.from('appbq-assets').upload(...)
- Obtener URL publica o firmada segun necesidad

8) Despliegue recomendado

Frontend:
- Netlify, Vercel o Cloudflare Pages.
- Publicar sitio estatico.

Backend:
- Supabase (no despliegas servidor propio).

Variables/secretos:
- Nunca exponer service_role key en frontend.
- Solo usar anon key en cliente.

9) Costos y viabilidad real

Si, es viable en plan gratis para:
- MVP
- Proyecto personal
- Primeros usuarios

Subir a plan pago cuando:
- Aumente trafico
- Necesites mas almacenamiento o rendimiento
- Requieras funciones avanzadas

10) Buenas practicas para no bloquearte luego

- Separa acceso a datos en funciones (ej: apiRegistros.js) y no mezcles todo en script.js.
- Define un contrato de datos claro (campos obligatorios/opcionales).
- Usa migraciones SQL versionadas.
- Agrega indices cuando crezca el volumen.
- Monitorea errores y cuotas del panel de Supabase.

11) Implementacion rapida en tu app actual

1. Agrega supabase-config.js.
2. Importa supabase en script.js.
3. Implementa login basico.
4. Crea/lista registros con RLS activo.
5. Despliega frontend y prueba flujo completo.

Resumen
- Para AppBQ, Supabase gratis es una opcion muy buena para arrancar rapido.
- Te evita mantener backend propio al inicio.
- Con RLS bien configurado, tienes seguridad base correcta.
- Puedes migrar a backend dedicado despues sin rehacer toda la UI.
